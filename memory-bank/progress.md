## What Has Been Decided

- The project direction is an AI-assisted game-master application.
- Canonical state stays outside the LLM.
- State updates are validated, persisted, and checkpointed.
- The project started oracle-first (Mythic-like deterministic backbone), but the next approved direction is a Cairn 2e-inspired rules layer built on top of that deterministic foundation.
- The LLM narrative engine must not roll dice, change chaos factor, or mutate state.
- Campaign opening content and oracle word banks are generated by the model at campaign initialization rather than hardcoded in domain defaults.
- The app uses LiteLLM for model switching, defaulting to OpenRouter Kimi K2.6 with task-based medium/high reasoning and deterministic fallback when no model/key is available.
- Runtime/model config is now explicitly two-layered: env-backed runtime defaults plus typed per-task profiles. Narration keeps its own env-facing creativity knobs, structured/mechanical calls stay low-temperature by default, the default timeout is now 600 seconds, and the old overloaded `LITELLM_MAX_TOKENS` semantics have been replaced by narrator-only `LITELLM_NARRATION_MAX_TOKENS` plus explicit task budgets in code.
- The UI is a bespoke Svelte 5 + Vite + TS frontend talking to a FastAPI backend; Streamlit was removed.
- The UI is chat-first: before play, a required character-creation flow chooses/templates/edits the protagonist; during play, character/inventory lives in a persistent left folio, mechanics live in a compact peek-on-demand inspector drawer with collapsible per-message receipts, slash commands (`/ask`, `/event`, `/scene`, `/chaos`, `/reset`, `/help`) are the explicit oracle entry points, and bare natural language goes to backend `/api/turn` for conservative automatic routing.
- Regenerate is now checkpoint-backed and preserves deterministic oracle outcomes; latest-response only. The client `Stop` button now triggers true server-side cancellation (`POST /api/requests/{id}/cancel`) before aborting the local fetch, so the LLM stops spending tokens on a turn the player abandoned.
- The backend now has a real cancellation path for streamed requests: `request_id`s are registered server-side, `POST /api/requests/{request_id}/cancel` flips a cooperative token, LiteLLM stream readers stop on cancellation, and streamed state mutations are discard-only until final commit.
- The app now has a real tiered memory sidecar. Committed canon is compacted into `data/memory.json`, and bounded planner/narrator memory context is drawn from that sidecar rather than replaying raw logs into prompts.
- The memory retrieval layer should avoid hardcoded lexical heuristics for semantic meaning. The implemented pass uses canonical references already present in outcomes/state, scene locality, active status, and recency ordering rather than stopword/token-overlap scoring.
- The first bug-fix pass on the memory/regenerate/planner failure issues is now implemented: streamed turns checkpoint pre-narration state, turn checkpoints persist `execution_context`, memory rebuilds use exact turn-checkpoint metadata, planner failure now surfaces explicitly at the API boundary, and setup-stream server aborts now surface as frontend errors instead of silent `null`s.
- The next backend latency pass is now implemented in the worktree: `GameService` emits typed NDJSON `stage` events before narration begins, continuity generation for `ContinuityUpdateScope.BOTH` now overlaps the expensive thread/NPC model calls while preserving deterministic apply order, and the pre-narration path now reuses one synced `MemoryState` snapshot instead of reloading the memory sidecar separately for thread, NPC, and narrator prompts. `ThreadUpdater` / `NPCUpdater` gained a split between generation and application specifically because shared-state mutation made naive parallel `update_*()` calls unsafe.
- Pixel font Alagard is the engine's visual voice (chaos numeral, dice, oracle tags, kicker labels, system messages); Cormorant Garamond stays the narrative voice; IM Fell English SC stays the display voice.
- Vanilla CSS (not Tailwind) is used so the "Oracle's Ledger" aesthetic is not homogenized.
- `.env` knobs are treated as developer / debug controls only; if a switch ever feels like it should be in the UI, that's a signal to promote it to a real control.
- The HTTP surface always returns the entire `GameState` so the frontend never reconciles partial diffs.
- The backend binds only to `127.0.0.1`; this is a personal-machine app, not for hosting.
- The chosen next mechanical direction is a **Cairn 2e-inspired dark-fantasy adaptation** rather than a pure oracle-only loop.
- The Cairn-inspired layer should stay **more structured than not**: canonical stats, HP, burden/slots, and deterministic item/mechanical effects, all kept outside the LLM.
- Existing authored characters should be preserved; the current authored character now has a one-time **LLM-backed** backend backfill path into nested Cairn mechanics state, and future mechanics-aware characters can bypass that path by submitting explicit mechanics.
- The frontend now surfaces the Cairn mechanical state **read-only**: the persistent left folio renders HP / STR / DEX / WIL, a segmented burden meter, fatigue, status chips (deprived, critically wounded, doomed, paralyzed, delirious, dead, overloaded), the LLM-derived skills/abilities lists, and per-item tag chips plus weapon-die / armor-bonus / uses / slots readouts. Equipped and primary-weapon badges sit on inventory items. The inspector exposes a collapsed "Cairn build notes" drawer showing the LLM's backfill rationale. The receipt strip recognizes the new `save` / `attack` / `harm` / `recovery` outcome kinds and renders structured Cairn fields (ability, target, success, damage-after-armor, hp/str/fatigue deltas, scar, rest kind, burden) in the body.
- The folio rail is a true two-column layout on most desktop windows: the rail width is `clamp(296px, 40vw, 720px)` so it grows aggressively into the gutter past the centered chat's 72ch cap, and `CharacterFolio` introduces an inner `.folio__layout` grid (the container query target) plus two `.folio__col` flex columns. The outer `.folio` element carries `container-type: inline-size` / `container-name: folio`; the inner layout exists because container queries can only restyle descendants of the container, not the container element itself. At `@container folio (min-width: 280px)` the rail flips from a 4-row single column into an identity (plate + condition / drive) on the left, ledger (mechanics readout + inventory) on the right. In two-column mode the condition / drive line clamp is dropped because the section now owns enough vertical room to show full text; the narrow fallback keeps the 3-line clamp so the inventory section doesn't get pushed offscreen. A vertical rule between the columns replaces the inter-section horizontal rules to avoid stranded lines above the shorter column's empty space.
- Backend streaming/persisted-thinking is now wired end-to-end for **all** model-backed outputs, not just active play. Character templates, setup quiz generation, scratch/template drafts, quizzed drafts, campaign start, player turns, generic actions, and narrative regeneration all share one NDJSON contract: a leading `meta` event with `request_id` + `route`, then `thinking_delta` / `content_delta` chunks, then a terminal `final_state` (state-mutating endpoints) or `final_payload` (setup endpoints, with a `kind` discriminator for `character_quiz` vs `character_draft`), or a terminal `error` event for backend-authored failures. Final thinking is preserved either in the setup response payload or on a persisted `GameEvent` when a `GameState` is committed, and is also surfaced on `final_state.thinking` for clients that don't want to re-read `state.action_log[-1].thinking`.
- Frontend pass complete. `web/src/lib/api.ts` exposes streamed siblings of every model-authored unary endpoint; `web/src/lib/store.svelte.ts` consumes them through `#runStreaming` (state mutations, with optional roll tumble between `mechanics_ready` and the first `content_delta`) and `#runStreamingPayload` (typed setup artifacts), each with a transparent fallback to the unary endpoint on a 404/405. ChatFeed renders a provisional DM bubble bound to `streaming.content` / `streaming.thinking` / `streaming.pendingOutcome`. The Inspector renders a default-open `Combat` drawer when an encounter is active, and `web/src/lib/combat.ts` adapts the canonical backend `EncounterState` (`state.encounter`) to the tracker's richer per-foe display shape (`status` derived from defeated/fled/critically_wounded; `morale_triggered` from the encounter's casualty / half-force flags).

## Known Issues and Risks

- The raw note contains exploratory language and assumptions that should be distilled carefully rather than copied verbatim into user-facing product copy.
- The oracle/rules layer should be modular from the start.
- Proprietary table content should not be copied into code unless licensing is handled.
- Browser testing observed expected 404s from the default local Ollama-compatible endpoint when no model server is running; the app handled this with fallback narration.
- The frontend now renders Cairn mechanics state read-only. It does not yet call the explicit Cairn mutation endpoints (`/api/cairn/attack`, `/harm`, `/recover`, `/equip`); those stay live but unbound until the explicit-controls pass.
- Live LLM playability: Kimi K2.6 reasoning latency dominates per-turn time. Routine narration is ~2-3 minutes, combat (with new encounter seeding) is ~3-5 minutes. Two pitfalls were identified and fixed empirically — `response_format={"type":"json_object"}` causes Kimi to over-deliberate (335s vs ~30s without), and `reasoning.effort` and `reasoning.max_tokens` are mutually exclusive on OpenRouter. We now pass only `reasoning.max_tokens` (per-task budgets: 600 router / 1200 encounter / 1500 routine narration / 3000 scene+random) and feed JSON output through `narrative.extract_json_object` (strips ```json fences, isolates first balanced `{...}`).
- The frontend `Stop` button now hits the backend cancel endpoint before aborting the local fetch (`web/src/lib/api.ts::cancelRequest`, wired via `store.cancelCurrentRequest`). Manual smoke against the live FastAPI server is still pending; the cancel POST contract is covered by `web/src/lib/cancel.test.ts`.
- The new memory layer is intentionally conservative about semantic selection. It now avoids hardcoded stopword/token heuristics, but may still want a richer semantic reranker later if long campaigns show recall gaps the current canonical/recency-bounded selection cannot solve.
- The persistence layer is still file-based and single-user. The bug-fix pass hardened atomic writes for checkpoints and turn checkpoints and stopped some forms of drift, but a full cross-file transaction boundary still does not exist in the same way a database transaction would.
- Product-design issue discovered during the B-01 inspection pass: the current visible NPC drawer is showing opener-seeded NPCs immediately, even when narration has not introduced them yet. Mechanically this is "working as built" (campaign generation returns exactly 2 NPCs and the Inspector renders `state.npcs`), but product-wise it is wrong: the player-visible roster should be an **introduced cast**, not an omniscient spoiler list. A follow-up backlog ticket now captures the desired contract: hidden backend cast allowed, but only introduced NPCs render in the Inspector, and recurring named antagonists in committed narration should be promoted aggressively into canonical records.
- Related UX issue: `Edit notes` currently works as a persistent `setting_notes` / `player_notes` editor, but the label reads like a generic scratchpad. The clarified desired role is narrower: this surface is only justified as a persistent OOC guidance / campaign-directives layer (stable steering that future generation should respect), not as a casual note-taking workflow the player is expected to use every session. A follow-up backlog ticket now captures that reframe.
- `F-16 Introduced-Only NPC Roster / Hidden Cast` backend is now landed. Canonical roster state is split: `GameState.npcs` now means the **introduced / player-visible** recurring roster, while `GameState.hidden_npcs` is backend-only continuity cast. `GameState.npc_roster_version` gates a one-time legacy repair path. Campaign bootstrap in `src/dungeon_master/campaign.py` now seeds opener NPCs into `hidden_npcs`, leaving the visible roster empty on fresh campaigns. `src/dungeon_master/npc_updater.py` still uses the bounded `create | update | retire` vocabulary, but each op now carries `player_visible: bool`, and the updater prompt sees the full canonical cast with visibility flags so hidden NPCs can be updated/reused without leaking into the visible roster. `GameService` adds two more invariants: (1) old saves missing the new roster split are repaired once on load and persisted with `npc_roster_version = 2`; (2) after narration commits, any hidden NPC whose exact name appears in the committed prose is moved into the visible roster and merged onto `OracleOutcome.referenced_npc_ids`. This means the current frontend, without any code changes, already stops showing opener spoilers because it still renders `state.npcs` and that list is now introduced-only.
- `B-02 Reframe Notes As Campaign Directives` backend is now landed too. `GameState` gained a dedicated `CampaignDirectives` model with `world_guidance` + `play_guidance`, and FastAPI exposes `POST /api/state/directives`. `GameService.update_directives(...)` persists those fields without appending a visible `action_log` event, because directives are durable OOC prompt steering rather than transcript-worthy canon. Prompt wiring landed in the key backend text-generation seams that still matter before the frontend pass: narrator prompt (`src/dungeon_master/narrative.py`), Cairn backfill/acquisition/encounter prompt builders (`src/dungeon_master/cairn.py`), NPC updater prompt (`src/dungeon_master/npc_updater.py`), and the OOC explainer (`src/dungeon_master/explainer.py`). The older `setting_notes` / `player_notes` fields remain intact for current campaign/backstory canon; this pass deliberately does **not** reinterpret them in-place because too much of the existing app still treats them as opening-world and character-facing text.

The product question that drove the shape was "what default behavior should the tray have?", and the user explicitly chose `prefill` over auto-submit and over a hybrid: clicking a tray pill edits the Composer buffer and the player still hits Send. That preserves the chat-first invariant — every action goes through the same player-typed turn the rest of the app already audits / receipts / memory-logs — and avoids inventing a parallel "button-driven" submission path that would duplicate slash and free-text routing.

`web/src/lib/common-actions.ts` is the new pure module. `deriveCommonActions(state, combat)` returns the visible tray strip in render order: `Ask oracle` (`/ask `), `Random event` (`/event`), `Scene check` (`/scene `), `Check gear` (prose `I take stock of what I'm carrying.`), and — only when `combat?.active === true` — `Attack` (prose `I attack `), `Recover` (prose `I take a short breather to recover.`), `Retreat` (`/retreat `). Order is contractual: the four baseline pills lead so the player's eye settles on stable left-aligned controls; combat-only pills tail so existing pills don't reshuffle under the cursor when an encounter begins. `applyCommonAction(buffer, action)` owns the buffer-replacement rule: empty / whitespace-only buffers are replaced wholesale; non-empty buffers preserve typed prose and append the prefill on a fresh line so the player sees what was added (caret lands at end-of-inserted-text in both branches). The append-not-clobber rule was deliberate — silent buffer replacement on a misclick would feel hostile, and the visible append plus caret-at-end gives an obvious one-keystroke undo (backspace).

`web/src/components/Composer.svelte` mounts the tray as a horizontal strip of pixel pills between the textarea and the send-row. The placement keeps the strip out of the slash-suggestion menu's float plane (which anchors to the textarea's top edge) and gives a natural left-to-right reading order: type → pick action (refine) → send. Clicks fire on `mousedown` (so the textarea doesn't blur first) and dispatch through `applyCommonAction`; the value write + `setSelectionRange(caret, caret)` re-focus is queued in a `queueMicrotask` so the bound write has propagated to the DOM before we touch the selection. Pills are `disabled` while `game.isLoading` and the toolbar carries `aria-disabled="true"` so screen readers pick up the lockout. The tray is structurally absent on the setup and ended layouts because the Composer itself is unmounted there (the ended layout swaps in `EndBanner`), so no archived-state special-casing was needed.

Two design calls that show up in code as "missing" features:

- `Check gear` deliberately maps to a planner-classified prose turn instead of an Inspector-open shortcut. The folio rail already shows the raw inventory at all times — the *useful* shape of "check gear" is the planner's `inspect_inventory` op narrating the carried items, not a duplicate readout of what the rail already displays.
- `Attack` and `Recover` are deliberately prose, not slash commands. There is no `/attack` / `/recover` parser entry today — F-11 tracks that work — and F-07 must not invent vocabulary the parser doesn't recognize. Routing them through the planner lets the same code path classify clicked-Attack and typed-"I swing" into the same Cairn `attack` route.

`F-09 History Browsing` is now landed (frontend-only). The product goal was "more usable long-session history browsing" with three candidate scopes: searchable transcript, scene/turn jump, and better oracle-history browsing. The implementation delivers the first and third; scene/turn jump was deliberately deferred because scene-index data lives in the backend's `memory.json`, not in `GameState`, and a clean frontend implementation needs either a memory-derived scenes API or a frontend approximation walked off `action_log` — neither was worth blocking the rest of F-09 on.

The transcript / oracle history pass is built around a new pure helper module `web/src/lib/history.ts`. `deriveTranscriptRows(state, notes)` mirrors `ChatFeed.svelte`'s message-derivation rules so the search surface and the chat surface can never disagree about what the transcript "is": oracle events fold into receipts under matching narrative rows (no duplicate hits per moment), narrative + player + system events render directly, the synthesized opening DM beat is included only when no narrative event exists yet (with a stable `opening_<state-id>` id matching ChatFeed's), and ephemeral client notes (slash help / errors) are flagged so search can opt in or out. `searchTranscript(rows, query, options)` does token-AND substring matching across row text *and* outcome summaries — so a search for "three of them" hits a receipt even when the prose phrases it differently — and returns chronologically ordered hits with pre-rendered snippets centered on the first matched token plus explicit highlight offsets so the renderer doesn't re-search. `findNarrativeEventForOracle(state, outcomeId)` returns the most recent narrative event linked to an outcome (regenerate produces a new narrative against the same outcome, so the latest match is the one to scroll to); null when nothing committed yet, which the inspector uses to gate the "Show in chat" affordance off rather than greying it out. `filterOracleHistory(history, filters)` AND-combines a free-text query (over summary, event-flavor fields, answer, scene status, and the Cairn `target_name` / `weapon_name` / `scar_result` / enemy-damage-source text — which is how the player actually remembers a combat moment, not "Harm 4 HP") with an `OracleKind` whitelist.

Cross-component scrolling is owned by the store. `web/src/lib/store.svelte.ts` exposes `scrollRequest: { eventId, seq } | null` plus `requestScrollTo(eventId)` and `consumeScrollRequest()`. The seq counter rotates on every call so back-to-back same-eventId requests still re-trigger the feed's scroll/flash effect — without a rotating seq, Svelte's `$effect` would see "the same value" and skip the re-run, which would mean clicking "Show in chat" twice in a row only worked the first time.

`web/src/components/ChatFeed.svelte` wraps each rendered message in an `.anchor` div carrying `data-event-id` (matching the synthesized `opening_<state-id>` for the first DM beat too). Auto-follow is opt-out: a 120 px bottom band drives `pinnedToBottom`, and the auto-scroll-on-new-token effect is gated on it. A floating "Jump to latest" pixel pill renders at the bottom-right of the chat shell whenever `pinnedToBottom === false`. Auto-follow uses *instant* scroll (`behavior: "auto"`) instead of smooth: a smooth-scroll mid-animation fires `onscroll` repeatedly with `distanceFromBottom > FOLLOW_THRESHOLD_PX`, which would briefly flip `pinnedToBottom` to false and stall the follow on the next streamed token. Smooth scroll is reserved for explicit jumps where the visual cue matters and there's no token race in flight. A `$effect` watches `game.scrollRequest` (guarded by a consumed-seq variable so cross-cutting reactivity doesn't replay a stale jump on remount), uses a `[data-event-id="…"]` selector inside the scroller to find the anchor, scrolls it into view (smooth, block: center), and applies a 1.7 s gold flash class with a reduced-motion fallback that does a static dwell tint instead.

`web/src/components/Inspector.svelte` adds a "Transcript" Drawer above "Oracle history" with a search input + Clear button, an empty-state hint when no query is typed, a no-matches hint, and a results list rendering speaker pill (`DM` / `You` / `Engine`), a `matched in receipt` tag when the hit was on an outcome summary, a relative timestamp, and the snippet with the matched substring wrapped in `<mark>`. Clicking a hit dispatches `requestScrollTo(rowId)` and closes the inspector — the player's goal at that point is to read the prose, not keep scanning. The pre-existing "Oracle history" Drawer gains an inline filter strip — text input + a row of nine `OracleKind` toggle pills (`Yes/No / Event / Scene / Action / Save / Attack / Harm / Recover / Retreat`) ANDed against the query. Each row pairs the existing `MechanicalReceipt` with a "Show in chat" pixel button that only renders when `findNarrativeEventForOracle` resolves; clicking it does the same scroll-and-close as the transcript hit. Filter / query state is component-local on the Inspector instance so it survives close/reopen within a session — hoisting to the store would force the same persistence question save slots will eventually answer, and "filters survive close/reopen, not page reload" matches what every other Inspector control already does.
